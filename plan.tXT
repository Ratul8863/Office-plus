OfficePulse вҖ” Real-Time Office Energy Monitoring System
Problem statement аҰ…аҰЁа§ҒаҰҜаҰҫа§ҹа§Җ boss chay office-er lights/fans live dashboard-e monitor korte, power usage аҰҰа§ҮаҰ–аҰӨа§Ү, and Discord bot diye status/usage ask korte. Deliverables-er moddhe high-level diagram, Wokwi/Tinkercad hardware schematic, simulated device data, real-time web dashboard, Discord bot, public repo, README, and demo video ache.

1. Main Goal
Amader goal:
Build a real-time office electricity monitoring system
where simulated hardware/circuit data goes to one backend,
and both web dashboard + Discord bot read the same live device state.
Core principle:
Single Backend = Single Source of Truth
Dashboard direct Wokwi/MQTT/MongoDB read korbe na. Discord bot-o direct Wokwi/MQTT/MongoDB read korbe na. Duita interface-i backend theke data pabe.

2. Problem
Office-e people lights/fans off na kore chole jay. Electricity bill barche. Boss chay:
1. Kon room-e kon device ON/OFF ache dekhte
2. Total power usage dekhte
3. Room-wise power usage dekhte
4. Office hour sesh hole alert pete
5. Discord theke quick status ask korte
6. Same data dashboard and bot-e reflect korte
Dashboard-er minimum features: live device status, live power consumption, room-wise breakdown, active alerts, and page refresh chara real-time update. Discord bot-er minimum commands: !status, !room <name>, and !usage; bot must use actual simulated data, hardcoded/random answer na.

6. Important Device Count Decision
Problem statement-e ekta inconsistency ache:
2 fans + 3 lights per room = 5 devices per room
3 rooms Г— 5 = 15 devices
But statement-e вҖң18 devices totalвҖқ bola hoyeche. Page-er office layout-o 2 fans + 3 lights per room show kore.
Best engineering solution:
Backend device config flexible rakhbo.
Default layout: 15 visible devices based on office diagram.
Compatibility mode: 18 devices support kora possible through config.
README-te clearly mention korbo this mismatch.
Code-e device.config.ts thakbe. Future-e judge jodi strictly 18 bole, just config update kore extra device add kora jabe.

9. Backend Responsibilities
device.store.ts
Current live device state memory-te rakhbe.
In-memory state = fastest live state
MongoDB = persistent history/current backup
telemetrySubscriber.ts
MQTT broker theke Wokwi ESP32 telemetry receive korbe.
telemetry.service.ts
Incoming data validate kore device state update korbe.
powerCalculator.service.ts
Total watt, room watt, active devices calculate korbe.
alert.rules.ts
Alert condition check korbe:
1. Device ON after office hours
2. All devices in a room ON for more than 2 hours
3. High power usage
4. MQTT device offline
socketServer.ts
Frontend-e live data broadcast korbe.
usageSnapshot.job.ts
Every 1 minute usage snapshot MongoDB-te save korbe.

21. Backend Core Logic
Power calculation
If device is ON:
   currentWatt = ratedWatt
Else:
   currentWatt = 0
Total power:
totalWatt = sum(all device currentWatt)
Room power:
roomWatt = sum(room devices currentWatt)
Estimated kWh:
kWh = wattHours / 1000
For MVP:
usage_snapshots every 1 minute
estimatedKwhToday = sum(snapshotWatt Г— intervalHours) / 1000

22. Alert Rules
Rule 1 вҖ” After office hours
If current time is after 5 PM or before 9 AM
and any device is ON
вҶ’ create alert
Rule 2 вҖ” Room fully ON too long
If all devices in a room are ON
and all have been ON for more than 2 hours
вҶ’ create alert
Rule 3 вҖ” High usage
If total watt > threshold
вҶ’ warning alert
Example threshold:
300W for demo
Rule 4 вҖ” Wokwi offline
If no MQTT telemetry from Wokwi for 60 seconds
вҶ’ device source offline alert

23. Simulator Strategy
Wokwi will represent one room. Other rooms can be simulated by backend.
Work Room 1 = Wokwi ESP32 telemetry
Drawing Room = backend simulator
Work Room 2 = backend simulator
This is best because:
1. Circuit connection demo hoy
2. Full office dashboard complete thake
3. Wokwi schematic simple thake
4. Backend still supports all rooms
Simulator logic:
Every 8 seconds:
   randomly choose Drawing Room or Work Room 2
   randomly toggle one device
   update lastChanged
   recalculate usage
   emit live update

28. Trade-offs
MQTT vs HTTP
MQTT better for IoT-style live telemetry.
HTTP simpler but less impressive for real IoT architecture.
Decision:
Use MQTT as primary.
Keep HTTP telemetry endpoint as fallback.
In-memory state vs MongoDB-only
MongoDB-only can be slower for real-time.
In-memory only loses history after restart.
Decision:
Use in-memory for live state.
Use MongoDB for persistence/history.
Wokwi all rooms vs one room
All rooms in Wokwi = complex and time-consuming.
One representative room = matches requirement and easier demo.
Decision:
Wokwi for Work Room 1.
Backend simulator for other rooms.

29. Final Product Summary
Amader solution:
OfficePulse is a real-time office energy monitoring system
that connects an ESP32-based Wokwi circuit to a Node.js backend using MQTT.
The backend maintains the single source of truth for device state,
stores history in MongoDB, calculates live power usage and alerts,
streams updates to a React dashboard through Socket.IO,
and powers a Discord bot through REST APIs.
This directly covers judging criteria:
Working real-time dashboard
Working Discord bot with same backend data
Good dashboard UX
Clear system diagram
Sensible Wokwi circuit schematic
Dynamic simulated data
Documented modular codebase
